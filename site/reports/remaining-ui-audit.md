# Fitmus 迁移后剩余页面 UI 审核清单（临时）

> 状态：待修复。本文档用于后续逐项整改；全部项目验收通过后可删除。
>
> 审核日期：2026-07-18  
> 审核对象：本地源码与 `https://fitmus-sport.pages.dev/` 线上部署  
> 审核基线：不追求像素级复刻；以内容完整、信息层级清楚、布局一致、桌面与移动端可用为目标。
>
> 分步执行计划：`site/reports/remaining-ui-repair-plan.md`

## 1. 审核范围与结论

本轮重点审核首页和 Bottom 之外的页面：首页与 Bottom 当前完成度较好，只作为共享视觉基线，不纳入本轮重做范围。

结论：资源迁移本身基本完整，当前最主要的问题是旧 WordPress/Avada Fusion HTML 被直接输出，但对应旧站网格、背景图、归档卡片和间距规则没有被新站接管。结果是图片文件虽然存在，页面仍会表现为空白、图片过大、内容挤在一起或结构层级消失。

本轮核查数据：

- 构建输出共 601 个 HTML 页面。
- 408 个产品和 176 篇新闻均有 `featuredImage`，且对应文件全部存在。
- 内容中共发现 805 个唯一 `/images/` 本地引用，缺失文件为 0。
- 408 个产品中，370 个图库只有 1 张图，36 个有 2 张，2 个有 3 张；“图库只有一张”不等于文件加载失败，但会产生内容单薄或疑似缺图的观感。
- 365 个产品正文仍含 `fusion-*` 旧站结构；368 个内容文件仍含旧 `data-bg-url` / 懒加载类属性。
- 四个主要静态页的旧结构量很大：About Fitmus 158 处 `fusion-*`；Fitmus Product 1,650 处；In the News 221 处；Contact Us 77 处。
- 线上抽查的六个代表 URL 均返回 HTTP 200，问题主要发生在呈现层而非路由层。

## 2. 优先级说明

- **P0**：页面主体明显失真或信息无法正常浏览，应最先处理。
- **P1**：共享组件或大量页面的一致性问题，影响主要使用路径。
- **P2**：可用但不整齐、内容密度或细节体验较差。
- **P3**：工具、维护性或非阻断性优化。

## 3. 问题清单

### UI-01（P0）四个主要静态页仍直接输出旧 Fusion 页面结构

影响页面：

- `/about-fitmus/`
- `/fitmus-product/`
- `/in-the-news/`
- `/contact-us/`

证据与表现：

- 四页均通过 `PageContent.astro` 将 `pages.json` 中的 `bodyHtml` 直接 `set:html` 输出。
- `PageContent` 当前只处理图片最大宽度、clearfix 和隐藏联系表单，未重建旧站列布局、背景图、卡片、标题分隔线和响应式规则。
- About Fitmus 在线桌面截图中，简介后出现接近一整屏的空白，再显示 “OUR START UP”；这是旧背景/容器高度仍在，但视觉内容未正常呈现。
- Fitmus Product 在线页面顶部直接显示 `All`、分类名、原始时间字符串，随后紧接超大产品图和裸文本；没有现代产品概览应有的分类卡片或整齐网格。
- In the News 在线页面把旧归档内容按超大图片、`Gallery`、标题、日期顺序直接堆叠，缺少统一新闻卡片结构。
- Contact Us 仍含旧地图/翻转卡片/隐藏表单插件结构，同时页面尾部又添加了新版地点卡片，存在两套内容体系。

建议：四页分别建立明确的 Astro 页面结构，只复用已迁移的文字和图片数据，不继续依赖旧 Fusion DOM 来承担排版。

进度：About Fitmus、Fitmus Product、In the News、Contact Us 已在 U1–U4 完成独立模板重构；产品详情与分类页已在 U5-A/U5-B 完成；新闻详情 U5-C 尚未执行。

验收标准：

- [x] 四页不再依赖 `fusion-*` 类名来实现关键布局。
- [x] 桌面端无异常大空白、负边距、浮动错位或超大图片占屏。
- [x] 390px 移动端无横向内容溢出，标题、正文、图片间距一致。
- [x] 页面内容完整，无为了修样式而误删原站的重要文字或图片。

### UI-02（P0）Fitmus Product 产品总览页信息结构失效

状态：已由 U2 完成总览页修复；分类列表和详情内容继续由 U5-A/U5-B 补齐。

影响页面：`/fitmus-product/`

问题：

- 页面不是八个一级产品分类的清晰入口，而是旧归档输出。
- 分类名、日期、图片、Gallery、产品标题和摘要没有卡片边界或统一间距。
- 桌面端首张图片高度接近整屏；移动端内容右侧被裁切，长分类/摘要造成横向溢出。
- 页面结构与顶部导航中的八个产品二级分类不一致，用户难以从总览页快速进入分类。

建议：改为八个分类卡片 + 每类简短说明 + 代表图；是否附带精选产品可在第二层呈现，不要在总览首屏直接铺旧归档全文。

验收标准：

- [x] 八个导航分类全部有入口，名称与 Header 完全一致。
- [x] 卡片图片比例、标题高度和间距统一。
- [x] 移动端单列，桌面端 3–4 列，无裁切或横向滚动。

### UI-03（P0）In the News 仍是旧归档而非新内容集合驱动列表

影响页面：`/in-the-news/`

问题：

- 当前路由读取 `pages.json` 的旧归档 HTML，没有直接使用已迁移的 176 篇 `posts` 内容集合。
- 页面显示旧式 Gallery 字样、超大主图、松散日期和标题，条目层次不清楚。
- 首页已经采用“最近 8 条”逻辑，但新闻总页与首页的数据来源和视觉规则不统一。
- 未来新增或修订文章时，旧归档 HTML 不会可靠地自动反映内容集合变化。

建议：以 `posts` 集合按日期降序生成新闻卡片列表；可保留全部文章并增加分页或“加载更多”的静态分页方案。

验收标准：

- [x] 首屏新闻按日期降序，卡片图、标题、日期、摘要对齐。
- [x] 卡片链接均指向现有文章详情路由。
- [x] 首页最近 8 条与新闻总页排序规则一致。
- [x] 不再显示 `Gallery` 等旧归档残留标签。

### UI-04（P1）顶部导航二级菜单图标基线和状态不统一

影响组件：`Header.astro`

问题：

- 桌面端和移动端都使用文本字符 `⌄`，但字号、行高和容器规则不同。
- 图标没有固定宽高、垂直居中盒子、打开态旋转或统一视觉尺寸，在线截图中符号明显偏低且偏小。
- 桌面下拉靠 hover/focus 显示，但没有可读的 `aria-expanded` 状态。
- 移动端导航采用横向滚动条；在 390px 截图中只能看到 “In the” 的一部分，Contact Us 完全在首屏外。
- 移动端 `<details>` 的绝对定位下拉处于横向 `overflow-x-auto` 容器内，存在下拉面板被裁切或定位不稳定的风险。

建议：用统一 SVG chevron 组件；固定 `size-4` 和 `shrink-0`，通过 `items-center` 对齐；移动端改成明确的菜单按钮/抽屉或纵向折叠菜单，不再依靠横向滚动承载全部一级导航。

验收标准：

- [x] 桌面与移动端使用同一 chevron 图标，文字与图标视觉居中。
- [x] 展开时图标旋转，键盘可操作，状态可被辅助技术识别。
- [x] 390px 宽度下五个一级导航均可访问，不需要猜测右侧还有内容。
- [x] 二级菜单不会被父级 overflow 裁切。

### UI-05（P1）产品、分类和文章详情页重复使用旧式 Sidebar 导航

影响页面：

- `/product/[slug]/`
- `/products/[category]/`
- `/[slug]/` 新闻详情

问题：

- 页面已有统一 Header，右侧又显示带第二个 Logo、八个产品分类和四个主导航链接的 Sidebar，形成两套导航体系。
- Sidebar 设计为左侧栏却放在页面右侧，组件使用 `border-r`，边界方向与实际位置不符。
- Sidebar 固定宽度 260px，但父级为三列网格；在中等桌面宽度下会压缩正文和产品卡片区域。
- 新闻详情右栏同时存在 ArticleDiscovery 与 Sidebar，信息密度过高。
- Sidebar 内部读取全部 products 集合但没有使用，增加无意义的数据加载。

建议：移除重复全站导航。产品详情可保留轻量的“产品分类/相关产品”模块，新闻详情只保留相关文章；共享全站入口统一由 Header 和 Footer 承担。

验收标准：

- [ ] 页面只保留一套全站导航。
- [ ] 详情页正文宽度稳定，右栏模块与页面语境相关。
- [ ] Sidebar 不再无用途读取 products 集合。

### UI-06（P1）产品正文大量保留旧 Fusion DOM 和内联样式

影响范围：至少 365 / 408 个产品。

问题：

- `product-content` 目前只规范图片宽度，没有处理旧列、标题分隔线、表格、浮动、内联宽度、隐藏节点和旧按钮。
- 同类产品详情的段落宽度、标题样式、列表和图片间距不一致。
- 旧 `data-bg-url` 或懒加载属性仍大量存在；若实际视觉只依赖这些属性，新站不会自动还原背景图。

建议：先建立受控的产品正文样式层，再分批清理最常访问的产品正文；不要一次机械删除所有 HTML，避免误伤产品参数和说明。

验收标准：

- [ ] `h2/h3/p/ul/ol/table/img` 在所有产品页有统一间距和可读宽度。
- [ ] 内联固定宽度、浮动和负边距不会破坏响应式布局。
- [ ] 有实际图片资源的背景内容改为语义化 `<img>` 或明确 CSS 背景。
- [ ] 隐藏/空插件节点不占据空间。

### UI-07（P1）“图片缺失”应按展示缺失与素材缺失分别处理

核查结论：本地路径完整，不应直接把全部问题归类为文件丢失。

具体问题：

- 370 个产品只有一张图库图片，详情页会显示大面积单图区域；这是素材数量不足或原站数据本就有限。
- 旧页面背景主要保存在 `data-bg-url` 和内联样式中，新页面未还原旧布局时会出现“应有画面但只剩空白”的展示缺失。
- 不同源图比例差异较大，而产品图库固定使用 4:3 容器和 `object-contain`，容易产生明显留白、主体大小不一致。
- 归档页直接显示原图，缺少统一裁切比例，导致一张图占据大半屏。

处理原则：

1. 先修复页面结构和图片组件；
2. 再列出确实需要新增素材的单图产品；
3. 不要通过拉伸、放大低分辨率图片来伪装素材完整。

验收标准：

- [ ] 归档卡片使用统一比例与 `object-cover`，详情主图按产品适合的 contain/cover 策略显示。
- [ ] 背景图未显示的问题全部有明确的结构替代方案。
- [ ] 单图产品允许保持单图，但界面不显示空缩略图位或误导性的“图库”控件。

### UI-08（P2）产品分类列表密度和内容层级不合理

状态：已由 U5-B 完成八个正式分类列表重构；`uncategorized` 保留为兼容路由，待业务确认归类。

影响页面：`/products/[category]/`

问题：

- 主内容只占三列中的两列，却在 `lg` 再分三张产品卡，实际卡片宽度偏窄。
- 分类页只有标题和“X products”，没有分类说明、面包屑或更清晰的选购上下文。
- 产品直接使用内容集合返回顺序，未显式定义稳定排序。
- `getStaticPaths()` 会按内容数据生成所有分类，但 Header 只列八个正式分类；若存在 `uncategorized`，导航和页面体系会不一致。

建议：移除重复 Sidebar 后使用更宽的 3–4 列网格；增加分类简介和稳定排序；明确处理 uncategorized 内容。

验收标准：

- [x] 卡片在常用桌面宽度下不过窄，标题不因列宽产生异常高低差。
- [x] 分类说明和产品数量具有清晰层级。
- [x] 构建结果顺序稳定，未分类产品不进入正式分类导航并保留兼容路由。

### UI-09（P2）文章详情页缺少完整的旧内容防护样式

影响页面：`/[slug]/`

问题：

验收标准：

- [ ] iframe/video 使用响应式容器。
- [ ] 表格可横向滚动且不撑破页面。
- [ ] 长链接、长标题和面包屑可换行，不触发页面级横向滚动。

### UI-10（P2）Contact Us 存在旧插件残留与新版内容重复

状态：已由 U4 完成静态页面清理；按项目要求不恢复联系表单交互。

影响页面：`/contact-us/`

问题：

- 原始 HTML 中仍有 5 个 `data-bg-url`、2 处 `wpcf7` 标记及旧地图/翻转卡片结构。
- 新站明确不需要联系表单交互，但当前做法只是 CSS 隐藏 `.wpcf7`，旧表单 DOM 仍保留。
- 新版地点卡片与旧联系信息可能重复，阅读顺序和间距不稳定。

建议：将需要保留的公司地址、电话、邮箱、地图链接直接写成干净的 Astro 结构，删除旧表单和插件占位节点。

验收标准：

- [x] 页面源码中不再输出 wpcf7 表单 DOM。
- [x] 地址、电话、邮箱只出现一次且链接可用。
- [x] 地图链接为普通外链，不依赖旧地图插件。

### UI-11（P3）构建完整性检查把合法 canonical 误报为旧站链接

影响脚本：`scripts/check-dist-integrity.mjs`

问题：

- `check:dist` 当前把构建时设置的正式域名 canonical（`https://www.fitmus-sport.com/...`）识别为 legacy host。
- 本轮对 601 个 HTML 页面产生 601 条问题，掩盖了真正的资源缺失或坏链接信号。

建议：允许 canonical、Open Graph、JSON-LD 中的正式站点绝对 URL；仍禁止正文、图片或导航意外回链到旧 WordPress 页面。

验收标准：

- [x] 正常构建时 `check:dist` 不因 canonical 失败。
- [x] 人为放入一个旧正文链接或缺失图片时，检查仍能准确失败并指出文件。

## 5. 推荐后续执行顺序

### U0：共享导航修整
